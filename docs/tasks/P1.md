### Knowledge ingestion pipeline

- описание: загрузка и обработка файлов/url/manual/obsidian источников.
- что должно получиться: управляемый ingestion со статусами и переиндексацией.
- критерии готовности (Definition of Done): источники обрабатываются через jobs и фиксируются в KB.
- зависимые задачи: Core API and UI-BFF skeleton (P0).
- ссылка на документацию: `/docs/architecture/MASTER_PLAN.md`, `/docs/database/SCHEMA.md`

### Hybrid RAG quality layer

- описание: vector+keyword retrieval, reranking, relevance filtering.
- что должно получиться: снижение нерелевантных ответов и галлюцинаций.
- критерии готовности (Definition of Done): quality gates проходят на тестовых датасетах.
- зависимые задачи: knowledge ingestion pipeline.
- ссылка на документацию: `/docs/runtime/CHAT_RUNTIME.md`, `/docs/architecture/SYSTEM_OVERVIEW.md`

### Knowledge OS UI

- описание: реализовать notes/links/graph интерфейс.
- что должно получиться: Obsidian-like управление знаниями в продукте.
- критерии готовности (Definition of Done): создание, редактирование и визуализация graph работают.
- зависимые задачи: knowledge ingestion pipeline.
- ссылка на документацию: `/docs/product/KNOWLEDGE_OS.md`, `/docs/frontend/ARCHITECTURE.md`

### Tool catalog and domain tools

- описание: каталог инструментов и бизнес-tools (lead/handoff/callback).
- что должно получиться: управляемые tool сценарии с аудитом.
- критерии готовности (Definition of Done): tool invocation отслеживается end-to-end.
- зависимые задачи: Tool runtime security (P0).
- ссылка на документацию: `/docs/runtime/CHAT_RUNTIME.md`, `/docs/api/API.md`

### Media manager

- описание: папки, файлы, usage map, reuse и выбор из UI.
- что должно получиться: централизованная медиа-система без raw URL.
- критерии готовности (Definition of Done): все ссылки на медиа идут через `MediaFile.id`.
- зависимые задачи: Core API and UI-BFF skeleton (P0).
- ссылка на документацию: `/docs/product/MEDIA_MANAGER.md`, `/docs/database/SCHEMA.md`

### Background jobs for indexing/sync

- описание: scheduler/workers для ingestion, sync и long runtime tasks.
- что должно получиться: устойчивое async выполнение с DLQ.
- критерии готовности (Definition of Done): failed jobs попадают в DLQ и доступны для replay.
- зависимые задачи: knowledge ingestion pipeline, tool catalog and domain tools.
- ссылка на документацию: `/docs/architecture/SYSTEM_OVERVIEW.md`
